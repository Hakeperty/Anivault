ARCHITECTURE NOTES
================================================================================

Design Pattern: MVC (Model-View-Controller)
- Models: IndexedDB stores (Library, Progress, Downloads, Settings)
- Views: Screen components with render() methods
- Controller: Router in main.js

Navigation Flow:
  Bottom Nav Click
    ↓
  Router.loadScreen(screenName)
    ↓
  Screen.render() → HTML
    ↓
  Screen.afterRender() → Event Listeners
    ↓
  Display in #screen-container

Data Flow:
  User Action → Event Listener
    ↓
  Screen method calls SearchCoordinator/DB
    ↓
  Scrapers return results OR DB returns stored data
    ↓
  Update component state
    ↓
  Re-render if needed

Error Handling:
- Try/catch in all async operations
- Toast notifications for user feedback
- Graceful fallbacks (MangaDex → MangaKatana)
- Empty state displays when data missing

KNOWN LIMITATIONS
================================================================================

1. CORS Issues:
   - HiAnime uses allorigins.win CORS proxy
   - Could be rate-limited if heavy usage
   - Alternative: Implement dedicated proxy server

2. HTML Scraping Fragility:
   - HiAnime parsing depends on HTML structure
   - Site updates will break scraper
   - Mitigation: Monitor logs, update selectors

3. Offline Support:
   - Currently no service worker
   - Can't play content without network
   - Mitigation: IndexedDB caching of metadata

4. Video Player:
   - HLS.js not yet integrated
   - Player screen is just a wrapper
   - Need HLS stream URL first

5. Download Manager:
   - No actual download implementation
   - UI only; needs Capacitor Filesystem
   - Need background task for queue
